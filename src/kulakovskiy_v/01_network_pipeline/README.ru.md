# Gene Network Pipeline

Предсказание генов-кандидатов ("guilt by association" через PPI-сеть) с
проверкой значимости через permutation-тест (случайные наборы генов),
сравнение с алгоритмом DIAMOnD и функциональная аннотация через DAVID.

Живёт на сервере: `~/work/pipeline` (см. `ssh 62.183.4.221`). Данные (сети,
списки генов) лежат в `/data/networks` и `/data/genes` — общие для всех, в
пайплайн не копируются. Сами результаты (`out1*`...`out6`, `analysis/`,
`pics/`) остаются в `~/work` (корень) и в этот репозиторий не копируются —
см. секцию внизу.

## Пайплайн

Три способа оценить/подтвердить кандидатов, все стартуют с общей
null-модели (шаги 1-4):

```
1 explore_network ──> common_nodes.txt (seed genes)
                             │
2 predict_candidates ────────┤──> out1-1/candidate-genes.txt  (реальный результат)
                             │
3 generate_random_sets ──────┤──> out2-1/rg-0.txt ... rg-9999.txt (10000 случайных seed-наборов)
                             │
4 run_null_model (parallel) ─┴──> out3-1/candidate-genes-0.txt ... -9999.txt (кандидаты для каждого случайного набора)
        │                                    │                              │
        │                        ┌───────────┴───────────┐                 │
        ▼                        ▼                       ▼                 ▼
  Ветка A: частотный тест   Ветка B: хабы/корр.   Ветка B: попарный p    Ветка C: DIAMOnD
  (шаги 5-7)                (шаг 8)                (шаг 9)               (свой алгоритм на common_nodes.txt)
        │                                                                    │
        └───────────────────────────── 10, 11: сравнение A vs C ─────────────┘
                                              │
                                        12: DAVID (аннотация)
```

### Шаг 1 — Разведка сети и затравочных генов

```bash
python3 scripts/01_explore_network.py \
    -n /data/networks/HS-PI.clean.tsv \
    -g /data/genes/human-orthologs-02.txt \
    -o common_nodes.txt
```
Печатает число узлов/рёбер сети и находит пересечение генов-ортологов с
сетью — чисто информационно, сколько из 865 orthologs реально есть в сети
(получилось все 865 из 865). Сохраняет их в `common_nodes.txt` и строит
`pics/degree_distribution*.png`.

### Шаг 2 — Предсказание кандидатов

```bash
python3 scripts/02_predict_candidates.py \
    -n /data/networks/HS-PI.clean.tsv \
    -g /data/genes/human-orthologs-02.txt \
    -o out1-1/candidate-genes.txt
```
На вход идёт исходный файл ортологов, а не `common_nodes.txt` — скрипт сам
отфильтровывает гены, которых нет в сети (см. `genes = [g for g in genes if
g in G]`), так что результат тот же, но именно так это было запущено в
реальном прогоне (проверено `diff` с `out1-1/candidate-genes.txt`).

Для каждого соседа seed-генов в сети считает, сколько разных seed-генов он
соединяет. Топ-100 (по умолчанию, см. `-t`) — кандидаты.

### Шаг 3 — Случайные наборы генов (null-модель)

```bash
python3 scripts/03_generate_random_sets.py \
    -n /data/networks/HS-PI.clean.tsv \
    -g 865 \
    -o out2-1/rg \
    -r 10000
```
`-g 865` — размер seed-набора (должен совпадать с реальным числом seed-генов
в `common_nodes.txt`, проверить: `wc -l common_nodes.txt`). Генерирует 10000
случайных наборов той же величины.

### Шаг 4 — Прогон шага 2 на всех случайных наборах

```bash
./scripts/04_run_null_model.sh \
    /data/networks/HS-PI.clean.tsv \
    out2-1/rg \
    out3-1 \
    10000
```
Параллельно (через `parallel`) вызывает `02_predict_candidates.py` на каждом
из 10000 случайных наборов → `out3-1/candidate-genes-{0..9999}.txt`. Это и
есть null-модель: распределение "какие гены оказались бы кандидатами просто
по случайности".

> Насколько точна эта null-модель зависит от числа реплик: с 10 000 реплик
> минимальное разрешимое p-value — `1/10000 = 0.0001`. Если ген встретился
> 0 раз, это значит только "p < 0.0001", не "p = 0" — увеличение до 20 000
> реплик лишь вдвое повышает разрешение и не гарантирует, что редкий ген
> вообще получит ненулевой счётчик. Для значений на порядки меньше
> (`1e-5` и ниже) permutation-подход в принципе не годится — там нужен
> аналитический тест (как в DIAMOnD, шаг ниже).

### Шаг 7 — Фильтрация значимых генов

```bash

    -i out5-1/candidate_gene.txt \

```

**Ветка B — сеть хабов и корреляций (шаг 8)**

```bash
python3 scripts/08_build_gene_network.py \
    -i out3 \
    -o analysis \
    -m 20 \
    -c 0.85
```
Строит бинарную матрицу "ген × реплика", считает статистику встречаемости,
попарные корреляции Спирмена, строит сеть из пар с корреляцией выше
`--corr_thresh`, считает центральности (degree/betweenness/closeness/
eigenvector/PageRank) и кластеры Лувена. Сохраняет в `analysis/`:
`matrix.csv`, `gene_statistics.csv`, `correlations.csv`, `centrality.csv`,
`clusters.csv`, `network.graphml`, `histogram.png`, `top_genes.png`,
`heatmap.png`, `network.png`.

Обратите внимание: по факту в прошлом прогоне этот шаг использовал `out3`
(1000 реплик), а не `out3-1` (10000) — можно направить на любую из двух
папок, результат не обязан совпадать.

**Ветка B — попарная значимость через гипергеометрический тест (шаг 9)**

```bash
python3 scripts/09_pairwise_significance.py \
    -i out3-1 \
    -o analysis/gene_summary.csv \
    -m 20 \
    -a 0.05
```
Для каждой пары часто встречающихся генов (>= `-m` реплик) проверяет,
встречаются ли они вместе чаще, чем ожидалось бы случайно (гипергеометрический
тест). Значимые пары агрегируются по гену: число значимых связей
(`n-edges`) и лучший p-value.

Шаги 8 и 9 отвечают на разные вопросы про одни и те же данные (структура
кластеров/хабов vs прямая попарная значимость) — оставлены оба, т.к. оба
реально использовались, но при желании можно ограничиться одним.

**Ветка C — DIAMOnD (задание преподавателя, отдельный алгоритм)**

[DIAMOnD](https://github.com/barabasilab/DIAMOnD) — стандартный алгоритм
"disease module detection": вместо простого подсчёта связей с seed-набором
(как в шаге 2) он на каждой итерации жадно добавляет ген с наименьшим
гипергеометрическим p-value связности с уже накопленным модулем.

```bash
# склонирован на сервере в ~/work/diamond/DIAMOnD (python2.7)
python2.7 external/DIAMOnD_with_pvalue.py \
    /data/networks/HS-PI.clean.tsv \
    common_nodes.txt \
    100 \
    out6/candidate-genes-with-pvalue.txt
```

> ⚠️ **Баг в оригинальном DIAMOnD.py**: сам алгоритм считает p-value на
> каждом шаге (`p = float(DIAMOnD_node_info[3])`), но в официальном коде
> эта строка нигде не пишется в выходной файл — сохраняются только `rank`
> и `DIAMOnD_node`. `external/DIAMOnD_with_pvalue.py` — патч в одну строку
> (добавлена запись `p` в выходной файл), ранжирование генов не меняется
> (проверено `diff` на первых двух колонках против исходного
> `out6/candidate-genes.txt`).

### Шаг 10 — Сравнение значимости: своя ветка A vs DIAMOnD

```bash
python3 scripts/10_compare_pvalue_diamond.py \
    --mine out5-1/candidate_gene_significant.txt \
    --diamond out6/candidate-genes-with-pvalue.txt \
    -a 0.05 \
    -o analysis/diamond_vs_mine_pvalue_comparison.png
```
Оба списка предварительно отфильтрованы по `p <= 0.05` (частотный p из
шага 7 и `p_hyper` из DIAMOnD). Строит диаграмму: слева — сравнение
p-value (лог-шкала) для генов, значимых **в обоих** методах, справа —
сводка по числу значимых генов и пересечению.

По текущим данным: 15 значимых генов в своём пайплайне, 100 — у DIAMOnD
(там *все* 100 выведенных узлов проходят порог, это ожидаемо — алгоритм
жадно останавливается на лучших p), 6 генов общие для обоих методов
(`1386`, `3932`, `867`, `6772`, `5335`, `4627`). У DIAMOnD p на 8-11
порядков меньше — это разные статистики (аналитическая связность с
модулем на каждом шаге агломерации против эмпирической частоты во всей
случайной permutation-модели), поэтому абсолютные величины некорректно
сравнивать напрямую — сравнение имеет смысл именно на уровне *какие гены
проходят порог* и *насколько уверенно*.

### Шаг 12 — Функциональная аннотация через DAVID

```bash
./scripts/12_run_david.sh "1386,867,3932,..." out_chart.html
```
DAVID (davidbioinformatics.nih.gov) заблокирован с GPU-сервера — скрипт

## История: что было исправлено по пути

- **`out5-1/candidate_gene.txt` содержал частоты больше 1** (например
  `9142.0001` вместо `0.9142`) — файл был получен более старой версией
  скрипта пересчёта частоты и с тех пор не пересчитывался, из-за чего
  фильтр `<= 0.05` не находил ничего. После пересчёта текущей (верной)
  версией шага 6 — 15 значимых генов вместо 0.
- **Оригинальный `DIAMOnD.py` считает p-value, но не записывает его в
  файл** — баг в апстриме (см. ветку C выше). Патч добавляет одну строку.
- Изначально было три параллельных bash-обёртки над `parallel` для запуска
  шага 4 на разных `out2*`/`out3*` — объединены в один параметризуемый
  `scripts/04_run_null_model.sh`.
- Было несколько более ранних/дублирующих черновиков подсчёта частоты и
  разведки сети — не сохранены отдельно, актуальная версия каждого шага
  описана выше.
